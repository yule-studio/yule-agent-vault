## 1. C10K (1999, Dan Kegel)

> "한 서버가 1 만 동시 연결을 처리할 수 있나?"

당시:
- select() FD_SETSIZE = 1024
- thread per connection = stack × N (메모리 폭증)
- context switch 폭발

해결 (2000s):
- **epoll** (Linux) / kqueue (BSD) — O(ready) 이벤트
- non-blocking I/O
- event loop (Nginx 1.0, Node.js, Redis)

자세히 → [[../io/select-poll-epoll]]

---

## 2. C10M (2013, Robert Graham)

> "1000 만 동시 연결"

C10K 의 해법 (kernel epoll) 도 한계:
- syscall overhead × N
- TCP stack overhead
- 메모리 (TCB) × N
- TIME_WAIT 누적

해결:
- **kernel bypass** — DPDK, AF_XDP, userspace TCP
- **Zero-copy** — sendfile, MSG_ZEROCOPY
- **io_uring** — syscall 최소화
- **SO_REUSEPORT** — worker per CPU
- **eBPF / XDP** — kernel 안 처리

자세히:
- [[../io/io-uring]]
- [[../io/zero-copy]]
- [[../linux/ebpf]]

## 3. 단계별 비교

| 동시 연결 | 일반 해법 |
| --- | --- |
| 100 | thread per connection (단순) |
| 1K | thread pool |
| 10K | **epoll + event loop** (C10K) |
| 100K | + tunign (sysctl, ulimit) |
| 1M | + multi-process / SO_REUSEPORT + 최적화 |
| 10M | **kernel bypass** (DPDK, XDP) |

대부분의 실무 = 10K-100K. 100K 까지는 잘 튜닝된 Nginx / Linux 면 충분.

## 4. Linux 튜닝 — C10K-100K 범위

### 4.1 ulimit

```bash
ulimit -n 1048576           # nofile
ulimit -u 65535              # nproc
```

systemd:
```ini
LimitNOFILE=1048576
LimitNPROC=65535
```

### 4.2 sysctl

```ini
# socket buffer
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216

# backlog
net.core.somaxconn = 65535
net.core.netdev_max_backlog = 65535
net.ipv4.tcp_max_syn_backlog = 65535

# TIME_WAIT
net.ipv4.tcp_tw_reuse = 1
net.ipv4.tcp_fin_timeout = 30

# keepalive
net.ipv4.tcp_keepalive_time = 300

# congestion
net.ipv4.tcp_congestion_control = bbr

# port range
net.ipv4.ip_local_port_range = 1024 65535

# fs
fs.file-max = 4194304
fs.nr_open = 1048576
```

자세히 → [[../linux/tuning]], [[../linux/networking]]

### 4.3 nginx config

```nginx
worker_processes auto;
worker_rlimit_nofile 1048576;

events {
    worker_connections 65536;
    use epoll;
    multi_accept on;
}

http {
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    keepalive_requests 100000;
}
```

## 5. SO_REUSEPORT — worker per CPU

```c
int opt = 1;
setsockopt(sock, SOL_SOCKET, SO_REUSEPORT, &opt, sizeof(opt));
bind(sock, ...);
```

같은 포트에 여러 worker 가 bind → kernel 이 load balance.
→ thundering herd 회피 + CPU 친화 + scaling.

Nginx 1.9.1+, HAProxy 1.8+.

## 6. event loop 패턴

```c
while (running) {
    int n = epoll_wait(epfd, events, MAX, timeout);
    for (int i = 0; i < n; i++) {
        if (events[i].events & EPOLLIN)  handle_read(events[i].data.fd);
        if (events[i].events & EPOLLOUT) handle_write(events[i].data.fd);
    }
    process_timers();
}
```

- 단일 스레드 (또는 코어 수)
- 모든 작업 비동기 / 짧게
- blocking call X (또는 thread pool 로)

## 8. 100K → 1M 으로

추가 기법:
- **io_uring** — syscall batch
- **kTLS** — TLS in kernel
- **MSG_ZEROCOPY** — large data
- **NUMA pinning**
- **interrupt coalescing**
- **IRQ affinity**
- **RFS / RPS / XPS** — packet to CPU 분산

## 9. C10M — kernel bypass

### 9.1 DPDK
NIC 의 사용자 공간 driver. kernel 우회.
- userspace TCP stack (mTCP, Seastar)
- 1 GB hugepage
- HFT, NFV

### 9.2 AF_XDP / XDP
eBPF + kernel bypass 의 hybrid.
- packet 처리 in kernel (eBPF)
- 사용자 공간으로 zero-copy

```c
// XDP program
SEC("xdp")
int xdp_drop(struct xdp_md *ctx) {
    return XDP_DROP;
}
```

Cloudflare 의 magic-firewall, Facebook 의 Katran.

### 9.3 Seastar / Glommio
사용자 공간 framework — share-nothing, per-CPU.
- ScyllaDB (Cassandra 대체)
- 100M+ ops/sec 가능

## 11. 자원 단위

연결당 RAM:
- TCB (kernel) ~ 4-10 KB
- 응용 buffer ~ 4-64 KB
- TLS state ~ 50-100 KB
- HTTP context ~ ...

1M 연결 = 수십 GB 메모리.

## 12. TIME_WAIT 함정

```
짧은 연결 (HTTP 1.0) 폭증 → TIME_WAIT 누적 → 새 연결 실패
```

해결:
- HTTP keep-alive
- `net.ipv4.tcp_tw_reuse=1`
- connection pool
- 짧은 `tcp_fin_timeout`

자세히 → [[../linux/networking]]

## 13. 면접 답 (요약)

> "C10K 는 epoll / kqueue 가 해결했고, 이후 C10M 은 io_uring / DPDK / XDP / kernel bypass + multi-queue NIC + SO_REUSEPORT 의 조합. 대부분 응용은 단일 호스트 epoll + tuning 으로 100K 까지 가능. 그 위는 분산 또는 kernel bypass."
