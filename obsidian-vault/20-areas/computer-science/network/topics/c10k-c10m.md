## 1. 한 줄 정의

**한 server 가 10K (또는 10M) 동시 connection** 처리할 수 있는가? — OS / 프로그래밍 모델 / 하드웨어의 진화 사례.

---

## 2. C10K — 2001, Dan Kegel

### 배경
- 1999-2001 — 인터넷 폭발적 성장
- 옛 서버 — 한 connection 한 thread/process (apache prefork)
- 1000 connection → 1000 thread → 메모리 폭발

### 문제
- Context switch 부담
- 메모리 (stack 8 MB × N)
- Lock 경쟁
- File descriptor limit

### 해결 — 이벤트 기반
- **select / poll** (옛)
- **epoll** (Linux 2.6+) / **kqueue** (BSD) / **IOCP** (Windows)
- **Non-blocking I/O**

---

## 3. 이벤트 모델

### select (옛, POSIX)
- 1024 fd 한계
- O(n) 매번 scan

### poll
- fd 한계 없음
- 여전히 O(n)

### epoll (Linux 2.6)
- O(1) — 활성 fd 만 반환
- edge-triggered / level-triggered
- 수만 connection OK

### kqueue (FreeBSD)
- 비슷 — fd / signal / timer

### IOCP (Windows)
- Completion-based (async I/O)

### io_uring (Linux 5.1+, 2019)
- 모던 — async I/O
- syscall 부담 ↓ (shared ring buffer)
- Network + disk

---

## 4. C10K 의 해결책 — 서버 사례

### Nginx (2004, Igor Sysoev)
- Event-driven (worker process + epoll)
- 한 worker 가 수만 connection
- Apache 의 prefork 모델 대체

### Node.js (2009, Ryan Dahl)
- libuv (epoll wrapper)
- JavaScript async — 한 thread

### Erlang / OTP
- BEAM VM — green thread (process)
- 한 server 수백만 process
- WhatsApp — 1 server 2M connection

### Go
- Goroutine (green thread)
- m:n scheduling
- 사용자 — sync 코드, runtime — async

---

## 5. C10K 가 풀려도 — 부족

### Disk I/O
- Network 빨라도 — disk 가 병목

### CPU
- 한 core — 100K req/s 한계

### Memory
- Connection 별 buffer (~10 KB) — 10M = 100 GB

### 결론
- C10K — OS / 모델 해결
- C100K — 가능 (튜닝)
- C1M — 일부 (Phoenix, Erlang, Go)
- C10M — kernel bypass

---

## 6. C10M — Robert Graham (2013)

### 정의
- 한 서버 1000 만 connection
- 일반 OS — 불가

### 해결 — Kernel Bypass
- DPDK (Intel)
- Netmap (FreeBSD)
- XDP / eBPF (Linux)
- DPDK + userspace stack (mTCP, Seastar)

### 사례
- Facebook Katran (eBPF / XDP)
- ScyllaDB (Seastar — shared nothing per core)
- Vector (DPDK)

---

## 7. DPDK — Data Plane Development Kit

### 정의
- Intel, 2010s
- Network packet 처리 userspace 에서
- Kernel network stack 우회

### 흐름
```
NIC → DMA → Userspace buffer (kernel 우회)
        → DPDK app (직접 처리)
```

### 장점
- 매우 빠름 (100M+ pps)
- Latency 매우 낮음

### 단점
- 코어 dedicated (busy loop)
- 일반 socket API 못 씀

### 사용
- 5G 네트워크
- 고성능 LB / 방화벽

---

## 8. XDP / eBPF — 모던 Linux

### XDP — eXpress Data Path
- Kernel 의 NIC driver 가까이
- Userspace 가능

### eBPF
- Kernel 안의 verified bytecode
- 안전하게 동작 (verifier)
- 다양한 hook (tracing, network, security)

### 사례
- Cloudflare LB
- Cilium (K8s networking)
- Facebook Katran

## 12. 함정

### 함정 1 — Thread pool 만 키우기
한계 — context switch 폭발. event loop 권장.

### 함정 2 — Single thread 의 한계
한 CPU core. Worker pool / multi-process.

### 함정 3 — TCP keep-alive 너무 자주
모든 connection 의 keep-alive → 대역 폭증.

### 함정 4 — Memory leak per connection
1 KB leak × 1M = 1 GB. 검사 도구.

### 함정 5 — TLS 의 부담
TLS handshake — CPU. Session resumption / hardware accelerator.

### 함정 6 — Sync DB driver
한 connection 에 sync DB call — block. Async driver / connection pool.
