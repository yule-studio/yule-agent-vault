## 1. 숫자

| 타입 | 크기 | 범위 / 정밀도 |
| --- | --- | --- |
| `SMALLINT` (`int2`) | 2 byte | -32,768 ~ 32,767 |
| `INTEGER` (`int4`) | 4 byte | ±2.1 billion |
| `BIGINT` (`int8`) | 8 byte | ±9.2 × 10^18 |
| `DECIMAL(p,s)` / `NUMERIC` | 가변 | 임의 정밀도 (돈 / 회계) |
| `REAL` (`float4`) | 4 byte | 단정밀도 부동소수 |
| `DOUBLE PRECISION` (`float8`) | 8 byte | 배정밀도 |
| `SMALLSERIAL` / `SERIAL` / `BIGSERIAL` | 자동 증가 | (구식 — 새 코드는 `GENERATED ... AS IDENTITY`) |

### 1.1 IDENTITY (SQL 표준, 권장)

```sql
CREATE TABLE users (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY
    -- 또는 GENERATED BY DEFAULT AS IDENTITY (override 가능)
);
```

vs 구식 `BIGSERIAL` — IDENTITY 는 표준 SQL + 더 명확한 제어.

### 1.2 돈 — `NUMERIC` 권장

```sql
amount NUMERIC(12, 2)   -- 999억까지 + 소수점 2
-- ❌ FLOAT — 부동소수 오차로 0.1 + 0.2 ≠ 0.3
```

## 2. 문자

| 타입 | 의미 |
| --- | --- |
| `CHAR(n)` | 고정 길이, 짧으면 padding |
| `VARCHAR(n)` | 가변, 제한 있음 |
| `TEXT` | 가변, 무제한 (실제 1 GB) |
| `CITEXT` (extension) | 대소문자 무시 |

### 2.1 권장: `TEXT` + `CHECK`

```sql
email TEXT NOT NULL CHECK (length(email) <= 320)
```

PostgreSQL 에서는 `VARCHAR(n)` 과 `TEXT` 의 성능 차이 거의 없음. `TEXT` + `CHECK` 가 더 유연.

## 4. 날짜 / 시간

| 타입 | 의미 |
| --- | --- |
| `DATE` | 날짜만 (4 byte) |
| `TIME` | 시간만 |
| `TIMESTAMP` | 날짜+시간, **timezone 없음** |
| **`TIMESTAMPTZ`** | ✅ 권장. UTC 저장 + 표시 시 변환 |
| `INTERVAL` | 기간 (`1 day 3 hours`) |

### 4.1 TIMESTAMPTZ 권장

```sql
-- 잘못된 예 — 서버 시간대에 따라 변동
created_at TIMESTAMP DEFAULT now()

-- 올바른 예
created_at TIMESTAMPTZ NOT NULL DEFAULT now()
```

### 4.2 자주 쓰는 표현

```sql
SELECT now();
SELECT now() - INTERVAL '7 days';
SELECT EXTRACT(YEAR FROM created_at);
SELECT DATE_TRUNC('day', created_at);
SELECT created_at AT TIME ZONE 'Asia/Seoul';
```

## 5. UUID

```sql
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";   -- 또는 pgcrypto

CREATE TABLE events (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid()  -- pgcrypto
);
```

### 5.1 UUIDv7 (시간 정렬)

PostgreSQL 17 부터 내장 (`uuidv7()`). 그 이전엔 extension.

```sql
SELECT uuidv7();   -- 시간 순 정렬 가능 — index locality 좋음
```

## 6. JSON / JSONB

| 타입 | 저장 방식 | 쿼리 속도 | 권장 |
| --- | --- | --- | --- |
| `JSON` | 텍스트 그대로 | 느림 | 입력 그대로 보관할 때 |
| **`JSONB`** | 바이너리 파싱 | 빠름 + 인덱스 | ✅ 대부분의 경우 |

### 6.1 사용

```sql
CREATE TABLE events (
    id    BIGSERIAL PRIMARY KEY,
    data  JSONB NOT NULL
);

INSERT INTO events (data) VALUES
('{"user": "alice", "items": [1,2,3], "meta": {"src": "web"}}');

-- 조회
SELECT data->'user' FROM events;       -- JSONB 값
SELECT data->>'user' FROM events;      -- 텍스트
SELECT data#>'{meta,src}' FROM events; -- 경로

-- 조건
SELECT * FROM events WHERE data->>'user' = 'alice';
SELECT * FROM events WHERE data @> '{"user":"alice"}';  -- 포함

-- 업데이트
UPDATE events SET data = jsonb_set(data, '{user}', '"bob"') WHERE id=1;
```

### 6.2 인덱스

```sql
-- GIN — 모든 키/값 가능
CREATE INDEX idx_events_data ON events USING GIN (data);

-- 특정 경로만
CREATE INDEX idx_events_user ON events ((data->>'user'));
```

## 7. Array

```sql
CREATE TABLE posts (
    id    BIGSERIAL PRIMARY KEY,
    tags  TEXT[] NOT NULL DEFAULT '{}'
);

INSERT INTO posts (tags) VALUES (ARRAY['postgres', 'tutorial']);

-- 조회
SELECT * FROM posts WHERE 'postgres' = ANY(tags);
SELECT * FROM posts WHERE tags @> ARRAY['postgres'];
SELECT * FROM posts WHERE tags && ARRAY['rust','go'];  -- 교집합

-- 인덱스
CREATE INDEX idx_posts_tags ON posts USING GIN (tags);
```

⚠️ Array 가 좋은 경우: 정렬 / 위치 의미 / 작은 컬렉션. 큰 컬렉션은 별도 테이블.

## 8. ENUM

```sql
CREATE TYPE order_status AS ENUM ('pending', 'paid', 'shipped', 'cancelled');

CREATE TABLE orders (
    id     BIGSERIAL PRIMARY KEY,
    status order_status NOT NULL DEFAULT 'pending'
);

-- 추가
ALTER TYPE order_status ADD VALUE 'refunded';
-- 삭제는 불가 (재생성 필요) → 신중하게
```

대안: `CHECK (status IN (...))` — 변경 자유. ENUM 은 표시 안정성 우선.

## 11. Range 타입

```sql
during TSTZRANGE       -- 시간 범위
price  NUMRANGE

INSERT INTO reservations (during)
VALUES ('[2026-05-13 10:00, 2026-05-13 12:00)');

-- 겹침 검사
SELECT * FROM reservations WHERE during && '[2026-05-13 11:00, 12:30)';
```

### 11.1 EXCLUDE constraint — 예약 충돌 방지

```sql
ALTER TABLE reservations
ADD CONSTRAINT no_overlap
EXCLUDE USING GIST (room_id WITH =, during WITH &&);
```

CREATE TYPE address AS (street TEXT, city TEXT, zip TEXT);

## 14. 타입 변환

```sql
SELECT '123'::INTEGER;
SELECT CAST('2026-01-01' AS DATE);
SELECT '{"a":1}'::JSONB;

-- 안전한 변환 (실패 시 NULL — PG 16+)
SELECT 'abc'::INTEGER;             -- 에러
SELECT to_number('abc', '999');    -- 0 (느슨)
```

## 15. 함정

### 함정 1 — `TIMESTAMP` (no TZ) 사용
서버 timezone 에 따라 의미 변함. 항상 `TIMESTAMPTZ`.

### 함정 2 — `MONEY` 타입
locale 의존, 잘 안 씀. `NUMERIC(12,2)` + 코드에서 통화 관리.

### 함정 3 — `CHAR(n)` 의 padding
짧은 값에 공백 padding → 비교 헷갈림. `TEXT` 사용.

### 함정 4 — `JSON` 을 `JSONB` 대신 사용
대부분 `JSONB` 가 정답. 입력 그대로 (key 순서, whitespace) 보존이 필수면 `JSON`.

### 함정 5 — 큰 컬렉션을 Array 에 저장
1000+ element Array 는 안티패턴. 별도 테이블 + FK.

### 함정 6 — `SERIAL` 의 시퀀스 owner 분리 문제
`SERIAL` 은 시퀀스를 자동 생성. 컬럼 삭제 시 시퀀스 남을 수 있음. `IDENTITY` 사용.
